## 1. 願景

### 1.1 一句話描述
> 武將養成型戰術 SRPG，兵種相剋為戰術骨幹，故事章節推進的西方奇幻長篇。

### 1.2 情感目標
玩家通關後留下：**一段故事**（角色、情節、結局令人回味）。

### 1.3 靈感來源

| 來源 | 借鑒重點 |
|---|---|
| 三國志（光榮）| 武將個性化、兵種相剋、武將特技 |
| 天使帝國（漢堂）| 方格 SRPG、美術品質、戰術深度 |
| 火焰之紋章（任天堂）| 章節制、支援對話、撤退機制 |
| Final Fantasy Tactics | 戰術深度（高低差暫不採用，保留可能） |

### 1.4 明確排除（不做）
- ❌ 4X 帝國經營（探索 / 科技樹 / 外交）
- ❌ 即時策略元素
- ❌ 大規模軍團（戰場 100+ 單位）
- ❌ Roguelike 隨機 build
- ❌ 多人聯機

## 2. 玩法核心

### 2.1 戰場規模
**中隊 6–10 人**出戰。陣中可招募武將約 8–12 名（最終遊戲 20–40 名）。

### 2.2 棋盤
方格制（具體格數待原型驗證，初期試 12×8 或 16×10）。

### 2.3 兵種系統
**5–6 種兵種，多元相剋**（非單純三角剋制）。

初步設想（**待原型實測調整**）：

| 兵種 | 攻擊範圍 | 特性 | 強剋 | 弱於 |
|---|---|---|---|---|
| 步兵（劍盾）| 近戰 1 格 | 平衡、高防 | 法師 | 槍兵 |
| 槍兵 | 近戰 1 格 | 對騎兵特攻 | 騎兵、步兵 | 弓兵 |
| 騎兵 | 近戰 1 格 | 移動力高 | 弓兵、法師 | 槍兵 |
| 弓兵 | 遠程 2–3 格 | 不可反擊 | 步兵、飛行 | 騎兵 |
| 法師 | 遠程 2 格 | 範圍魔法 | 騎兵、密集 | 步兵 |
| 治癒 | 輔助 1 格 | 補血、buff | (無剋制) | (脆) |

> 💡 數值與相剋係數需透過原型反覆迭代，不要在紙上算到完美。

### 2.4 戰鬥隨機性
有命中率/閃避（火紋風），但偏向**高命中、低變異**（避免運氣決定戰局）。
- 基礎命中 80–95%
- 暴擊率 5–15%
- 兵種相剋影響傷害倍率（×1.3 或 ×1.5）而非命中率

### 2.5 養成系統
**等級 + 裝備為主，無職業樹**。

- **等級**：1–20，數值線性成長
- **裝備**：武器、防具、飾品（各槽位 1 件）
- **武將特技**：每名武將有 1–2 個獨特特技（固定，不可學/不可換）
  - 例：「火計」「奮起」「先制」「庇護」「狂戰」
- **羈絆系統**（待設計）：特定武將相鄰有 buff（同陣營、同鄉、師徒）

### 2.6 死亡機制
角色 HP 歸 0 → **撤退**（**非永久死亡**）。
- 戰鬥中：被撤退的角色該場無法再上場
- 戰鬥後：HP 帶到下一場（需在據點補血/休整）
- 主線特定戰役：劇情角色死亡可能觸發 BAD ENDING（保留設計空間）

## 3. 結構與節奏

### 3.1 章節制循環
```
劇情過場 → 據點 → 戰場戰鬥 → 結算 → 劇情過場 → ...
```

### 3.2 據點功能（中介層）
| 功能 | 內容 |
|---|---|
| 商店 | 買賣武器、防具、消耗品 |
| 招募 | 解鎖新武將加入陣營 |
| 支援對話 | 武將間累積好感解鎖對話（火紋風） |
| 整備 | 裝備配置、隊伍編組 |

### 3.3 時長
- **單場戰鬥**：45–60 分鐘
- **總關卡**：30+ 關（長期目標，分階段擴充）

## 4. 美術方向

### 4.1 風格
**像素藝術棋盤 + 立繪**（火紋 GBA / 天使帝國 II 風格）

### 4.2 資源策略（混合）
| 元素 | 來源 |
|---|---|
| 立繪 | AI 生成 → 人工修圖統一風格 |
| 像素 sprite | 付費資源包（Itch.io / OpenGameArt）|
| UI / icons | AI 生成 |
| BGM / SFX | AI 生成（Suno、Udio）或購買授權 |

### 4.3 商業化考量（如未來上架）
⚠️ AI 內容必須揭露（Steam 政策）。SRPG 受眾對美術要求高，全 AI 路線在商業化前需重新評估。

## 6. 技術選型

| 項目 | 選擇 | 理由 |
|---|---|---|
| 語言 | TypeScript | Python 出身者學習友善（type hints 概念一致）|
| 遊戲框架 | Phaser 3 | Web 2D 事實標準，文件與社群豐富 |
| 打包 | Vite | 啟動快、零配置、現代 Web 標配 |
| 編輯器 | VS Code | 免費、TS 體驗最佳 |
| 版本控制 | Git | Phase 1 開始 |
| 部署 | itch.io（試玩階段）| Steam 等 Phase 2 後評估 |

### 6.1 不選的原因
| 方案 | 為什麼不 |
|---|---|
| Godot 4 | 個人偏好 Web 為主、share 連結能玩 |
| PixiJS（直接用）| 太底層，新手不友善 |
| Unity | 商業授權與 Web 體積問題 |
| React + Canvas 混搭 | 對遊戲渲染流程沒幫助 |

## 7. 開發策略

### 7.1 漸進式垂直切片

| Phase | 範圍 | 目標 |
|---|---|---|
| 0 | 學習 + 設計 | TS/Phaser 基礎 + 設計文件 |
| 1 | 微型原型 | 1v1 戰鬥能跑（無美術、無故事）|
| 2 | MVP 垂直切片 | 序章 + 3 關，完整核心循環 |
| 3 | 第一幕 | 8–10 關 |
| 4+ | 擴充 | 朝 30+ 關前進 |

**核心原則**：每個 Phase 結束都是**完整可玩產品**，不留半成品。

### 7.2 心態原則
- **不設死線**
- **每階段交付完整可玩**
- **商業化是 Phase 2 後再評估的選項**
